# T1 時光機頁面「擁擠問題」解法討論

> 給夥伴一起看、一起選的討論文件。
>
> 背景文件：`docs/constitution-features/_archive/010-t1-timeline-crowding.md`（完整的 ideation + review 紀錄）

---

## 問題是什麼？

**簡單版**：T1（`/past` 頁面，憲法時光機）現在放 10 筆釋字，使用者要滾 19 個螢幕高度才能看完一輪。SSOT 試算表持續成長，預估未來會到 30 筆以上 — 那時候頁面會變成「永遠滾不到底」。

**技術版**：
- 頁面 scroll 容器高度 = `entries 筆數 × 100vh`（每筆釋字要佔 1 個螢幕高度來做 sticky 動畫）
- 現在 10 筆 = 1000vh（1920px 螢幕要滾 ~19 次螢幕高度）
- 未來 30 筆 = 3000vh，40 筆 = 4000vh
- 除此之外，中央的年份標籤會隨筆數增加而疊字不可讀；全部 DOM 節點一次載入，首屏成本線性上升
- 沒有任何「跳到 2010 年」的導航 affordance；也沒有 filter / search

**核心結構問題**：所有 entries 擠在同一條 scroll 軸上，沒有任何「escape hatch」。

---

## 五個候選方案總覽

| 代號 | 名稱 | 改動規模 | 解什麼問題 | 不解什麼問題 |
|---|---|---|---|---|
| **A** | 年代折疊 | 中 | 拆分 scroll 軸、減少 DOM、解決年份疊字 | 需要多一次點擊破壞沉浸感 |
| **B** | Filter chips | 小 | 給使用者一條逃生路（按分類／年代過濾） | 不下 filter 時依然 1000vh |
| **C** | Timeline mini-map | 中 | 任意年代快速跳轉 | scroll 距離不變、mobile 難做 |
| **D** | 分年代子頁 | 大 | 終極 scaling、URL 可分享 | 破壞「一頁看完憲法史」敘事連貫 |
| **E** | 搜尋 / Cmd+K | 小–中 | 精準查詢（釋字號、關鍵字） | 不知道要找什麼的使用者幫不到 |

---

## 方案 A — 年代折疊（Decade Sections）

### 使用者體驗

進入 `/past`，先看到短短的 hero section，然後是 4–5 張年代卡片：

```
┌─────────────────────┐  ┌─────────────────────┐
│  1990s              │  │  2000s              │
│  3 則釋字           │  │  2 則釋字           │
│  民主轉型 ·         │  │  正當程序 ·         │
│  言論自由           │  │  遷徙自由           │
└─────────────────────┘  └─────────────────────┘
┌─────────────────────┐  ┌─────────────────────┐
│  2010s              │  │  2020s              │
│  2 則釋字           │  │  3 則釋字           │
│  原住民權 ·         │  │  性別平等 ·         │
│  性取向平等         │  │  團結權             │
└─────────────────────┘  └─────────────────────┘
```

點擊某個年代卡片 → 該 section 就地展開，進入熟悉的 sticky 雙欄動畫，但 **只有那個年代的 2–3 筆釋字**（scroll 距離變成 200–300vh 而非 1000vh）。看完年代末有「下一年代 →」按鈕。

### 技術複雜度
**中**。主要改動集中在 `src/app/past/page.tsx` 頂層結構（從單一 scroll 容器改為多個折疊 sections），每個 section 內部可沿用現有 sticky + IntersectionObserver 邏輯。資料層不變（在 component 內做 `groupBy(decade)` 分組）。

### 優點
- 直接解決「所有釋字擠在同一 scroll 軸」的核心結構問題
- 敘事合理：年代本身就是故事單位（90 年代民主轉型 / 2010 年代性別平權）
- 規模化友善：筆數成長自然分攤到不同年代（20 筆 → 每段仍只有 4–7 筆）
- 大幅保留現有視覺資產，section 內部不動美術
- 手機體驗順：折疊 section 是標準 mobile pattern

### 缺點
- 使用者必須多點擊一次才進入內容，破壞現在「開頁即沉浸」的無縫感
- 某些年代可能只有 1–2 筆（1990s 多、2000s 少），四張卡片視覺不均
- hero → 第一個 section 的銜接需要設計（例如要不要預設展開第一個年代？）

### 類似產品
Apple 官網發表會年表、維基百科年表收合、紐時長篇報導的章節折疊

## 方案 B — Filter Chips（最小改動）

### 使用者體驗

進入 `/past`，看到熟悉的 hero + sticky scroll 體驗，**頂部 sticky 一列 chip**：

```
[ 全部 10 ]  [ 言論自由 2 ]  [ 性別平等 2 ]  [ 1990s 3 ]  [ 2010s 2 ]
```

預設「全部」= 完全跟現在一樣的體驗。點 `性別平等` → 畫面只 render 2 筆性別平等相關的 entries（scroll 容器自動縮到 200vh），一樣跑 sticky 動畫但只針對這 2 筆。支援多選、有「清除」按鈕。

### 技術複雜度
**小**。只改 `src/app/past/page.tsx`：加一個 `useState<Set<string>>(filters)`，把 `HISTORY_DATA` 先 filter 再 map。Chip UI 可複用現有 Tailwind tokens，無須新組件。需要小心處理 `IntersectionObserver` 在 filter 變化時的重新初始化。

### 優點
- 改動面最小、風險最低 — 1–2 天可上線
- 完全保留現有動畫與美術
- 對「我只關心性別平等歷史」的使用者超友善
- 可以跟其他方案共存（未來疊加 A 或 E 都可）

### 缺點
- **沒解決核心問題**：不下 filter 時仍然是 1000vh scroll
- 只是多給一條逃生路，B 對「瀏覽型」使用者（其實是擁擠問題的主受害者）幫助有限
- 隨著筆數成長，chip bar 自己也會變擠（10 個 category + 4 個年代 = 14 個 chip）

### 類似產品
Medium tag filter、Pinterest board filter、LinkedIn 的 "Top / Recent" toggle

## 方案 C — Timeline Mini-map（側邊跳點）

### 使用者體驗

進入 `/past`，同樣的 sticky scroll 體驗，但螢幕右緣永遠有一條 40px 寬的直立 mini timeline：

```
           │ ● 1990     ← hover 時顯示「野百合學運 · 釋字第 261 號」
           │ ○ 1991
           │ ○ 1994
           │
           │ ○ 1999
           │
           │ ○ 2003
           │ ○ 2004     ← 目前在此（高亮）
           │
           │ ○ 2017
           │ ○ 2022
           │ ○ 2023
```

每筆釋字一個 dot，目前位置自動高亮。**點擊任一 dot → smooth-scroll 瞬間跳到該年代的 entry**，強迫自己從頭滾的時間大幅縮短。

### 技術複雜度
**中**。新增一個 `<TimelineMinimap>` fixed-positioned 組件，接收 `entries` + `activeIndex` 作為 props。點擊時用 `window.scrollTo({ top: triggerElement.offsetTop, behavior: 'smooth' })`。Mobile 需要另外設計（可能折疊成 drawer 或 FAB）。

### 優點
- 保留現有 scroll 敘事，純粹加上「地圖」
- 「我在哪裡？」的空間感永遠可見
- 一鍵跳到任意年代（從 ~20 秒滾動壓到 ~2 秒）

### 缺點
- **不縮短 scroll 距離**、**不減少 DOM** — 只是讓長度變得「可跳躍」
- Mobile 體驗差（右側沒空間）
- 與頁面中央現有的 timeline 視覺功能重疊，可能讓使用者困惑「哪一條才是主的？」

### 類似產品
MDN 長文右側 outline、iOS 聯絡人 A-Z 索引條、Figma 文件大綱 panel

## 方案 D — 分年代子頁（Route-level Split）

### 使用者體驗

進入 `/past` 看到一個 **overview 頁 — 不再有 scroll 動畫**，而是 4–5 張精美的年代卡片（每張有代表性的歷史背景圖）：

```
┌──────────────────────────────┐
│  [背景：野百合學運歷史照]    │
│                              │
│  1990s                       │
│  民主轉型的時代              │
│  3 個重要釋字                │
│                              │
│  [進入 →]                    │
└──────────────────────────────┘
```

點擊 `1990s` → 導航到 `/past/1990s`。那個子頁才是熟悉的 sticky 雙欄 scroll 體驗，但只有 3 筆 entries（300vh）。每個子頁有清楚的 URL，可深連結。

### 技術複雜度
**大**。要新增 `src/app/past/[decade]/page.tsx` 動態路由 + `generateStaticParams`，把現有的 T1 敘事程式碼搬進子頁。`/past` 本身變成 overview，**需要新的美術設計**（4–5 張有 hero 圖的年代卡片）。影響所有連結到 `/past` 的地方（導覽列、其他 tracks）。

### 優點
- 終極 scaling：未來成長到 100 筆也不會崩
- 每個子頁都很輕（300vh 以內）、DOM 節點少
- URL 可分享、SEO 友善（`/past/2010s` 可以直接轉貼）
- 年代 overview 本身可以是視覺亮點（用歷史照當背景）

### 缺點
- **破壞「一頁看完整部憲法史」的情感張力** — T1 的 UX 敘事很依賴從 1990 一路滾到 2023 的累積感
- 改動面最大：新路由、新 overview 頁、連結更新、分享連結結構
- **需要新美術設計才能做**，不是純工程任務
- 超出「趕快解決擁擠」的範疇，比較像 T1 的 v2 重做

### 類似產品
維基百科「21 世紀年表」vs「20 世紀年表」分頁、Apple Newsroom 按年份分的 archive

## 方案 E — 搜尋 / Cmd+K（精準查詢）

### 使用者體驗

進入 `/past`，同樣的 sticky scroll，頁首多一個搜尋欄（或 Cmd+K 彈出 palette）：

```
[ 🔍 搜年份、釋字號或關鍵字（如：同婚）______________________ ]
```

輸入 `同婚` → 下拉建議 `2017 · 釋字第 748 號 · 同性婚姻`。點選 → smooth-scroll 到那筆 entry，著陸時閃爍高亮提示。輸入 `748` → 直接跳釋字 748。輸入 `2017` → 顯示 2017 年所有 entries。

### 技術複雜度
**小–中**。加 `useState(query)` + 簡單的字串比對（title / category / ruling_id / year）。可選擇加 `fuse.js` 做 fuzzy match（多一個 dep，但現在 10–30 筆其實用不到）。004 的 T2 搜尋有類似組件可參考。

### 優點
- 對「記得釋字 748」這類精準查詢使用者最佳
- 改動小、風險低
- 與其他任何方案都能共存

### 缺點
- **對「不知道要找什麼、只想瀏覽」的使用者沒用** — 而這些人才是擁擠問題的主要受害者
- 現階段資料規模（10–30 筆）其實搜尋必要性不高（大砲打蚊子）
- 隨資料長大價值才會顯現

### 類似產品
Linear Cmd-K、Notion 頁內搜尋、GitHub issue search

## 方案組合

### A + E（年代折疊 + 搜尋）— **ideation 推薦組合**

- **瀏覽型使用者** 走 A 的年代折疊敘事（「我來逛逛 90 年代的憲法發展」）
- **目標型使用者** 走 E 的搜尋直接跳點（「我要找釋字 748」）
- 額外複雜度：**小**（E 本身改動就小，疊在 A 之上只需讓搜尋結果自動展開該年代 section）
- 回應了兩種互斥的使用者意圖，A 單獨只服務前者

### A + B（年代折疊 + filter chips）— 不推薦

年代本身已經是一種 filter；再疊 category chip 會有兩層過濾邏輯，UI 容易混亂。**除非未來 category 也爆炸**（例如 20+ 個類別），否則不值得。

### C（mini-map）+ 其他任何方案 — 不推薦

一旦走 A 或 D，頁面就已經分段／分頁了，mini-map 的價值大幅下降，資訊架構重複。

---

## 推薦方案（ideation 原始建議）

**推薦 A（年代折疊）作為主方案，搭配 E（搜尋）作為組合。**

### 為什麼選 A

1. **直接對應問題核心**：擁擠的根源是「所有 entries 疊在同一條 scroll 軸」；A 是唯一從「分割 scroll 軸」層面動刀的方案。B/E 只加 escape hatch、C 只加地圖、D 雖也分割但代價太大
2. **敘事合理**：年代本身是故事單位，分塊不破壞時光機情緒曲線，反而強化章節感
3. **規模化可控**：entries 從 10 長到 30、40 時，壓力線性分攤到 4–5 個年代（每段 6–10 筆），仍在現有 sticky scroll 的甜蜜點內
4. **保留現有視覺資產**：section 內部原封不動，不需重做美術
5. **複雜度中等可控**

### A 放棄了什麼（誠實）

- vs B：B 改動更小；若優先順序是「盡快上線」B 更划算
- vs D：D 對超大規模（50+ entries）表現更好
- vs C/E：沒有「任意快速跳點 / 精準查詢」能力 — 所以要搭 E 補齊
- 共同代價：引入「需點擊展開」這一步，破壞「開頁即沉浸」的無縫感

## 給討論會議的決策框架

把優先順序依序問一次：

| 如果你最在乎的是... | 就選 |
|---|---|
| **盡快上線、之後再說** | **B**（filter chips） |
| **真正把核心 UX 修好** | **A**（年代折疊） |
| **為未來成長到 100 筆以上做準備** | **D**（路由分割） |
| **不動敘事只加導航輔助** | **C**（mini-map） |
| **直接跳到釋字 748 的能力** | **E**（搜尋） |
| **核心修好 + 進階使用者捷徑** | **A + E**（組合） |

---

## 需要團隊一起決定的事

1. **選哪個方案？**（A / B / C / D / E / A+E / 其他組合）
2. **上線時程有多緊？** — 會影響 B vs A 的取捨
3. **T1 長期會成長到多少筆？** — 會影響要不要 D
4. **年代是不是真的是最有敘事感的分段方式？**
   - 或者「主題」更強（民主轉型 / 性別平權 / 原住民權）？
   - 或者「憲法條號」？
   - 如果不是年代，A 的設計邏輯要調整
5. **如果選 A，第一個年代要預設展開嗎？**
   - 預設展開 → 部分保留「開頁即沉浸」的感覺
   - 預設收合 → 一致性，但使用者第一次進來看到四張卡片可能不知道要做什麼
6. **如果選 A+E，搜尋命中時怎麼處理收合 section？**
   - 自動展開目標 section → 流暢但會改變頁面狀態
   - 跳出來提示「這筆在 2010s，點此展開」→ 保留使用者的瀏覽狀態

## 附錄：目前 T1 實作的重點資訊

（工程夥伴用，非工程夥伴可略過）

- 入口：`src/app/past/page.tsx`
- 資料：`src/data/history.json`（10 筆，年份 1990–2023，10 個不同 categories）
- Scroll 容器高度：`HISTORY_DATA.length * 100vh`
- Active entry 偵測：`IntersectionObserver`，`rootMargin: "-45% 0px -45% 0px"`（畫面中央 10% 觸發）
- 左右雙欄動畫：絕對定位疊放 + `.textbook-item.active/.exit-up/.exit-down` CSS class
- 中央 timeline 年份標籤：`whitespace-nowrap`，flex `justify-between` 均分 60vh 區段
- 背景圖：每筆 reality-item 各自載入一張 Unsplash 2070w 圖（首屏成本隨 N 線性上升）
